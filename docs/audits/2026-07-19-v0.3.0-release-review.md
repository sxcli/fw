# Review: sxcli-fw v0.3.0 (read-only, Fable agent, 2026-07-19)

`go test ./...` and `go vet ./...` both PASS (all 8 packages). Findings ranked; #1–#5 confirmed empirically with probe binaries against the local tree.

## Findings

**1. CRITICAL — main.go:314-319 + introspect.go:157 — `Introspector.Arguments` panics (nil-pointer dereference) when `args` contains `--upgrade-config`.**
`plan()` returns early with `p.upgrade = true` and `p.sch == nil`, the collector is clean, so `Arguments` takes the happy path `out = argInfos(p.sch)` and `argInfos` calls `sch.HelpSections()` on a nil `*engine.Schema`. Reproduced: `Arguments("probe", []string{"--upgrade-config"})` → PANIC. Contradicts spec §4 (planning-token flags "are inert data"); since applet panics are deliberately not recovered (§5), a completion consumer completing `mytool --upgrade-config --config <TAB>` crashes the whole binary. `--write-config`/`--help` ARE pinned inert by `TestArgumentsIsSideEffectFree`; `--upgrade-config` was missed.

**2. MAJOR — main.go:430-451 (`upgradeConfig`) — the whole-catalog upgrade schema applies closure-scoped uniqueness rules, making `--upgrade-config` unusable for legal multi-applet binaries.**
The spec makes duplicate long names an error *across the closure* (§6), so two applets with disjoint closures may each declare `conf:"port"` — such a binary runs fine. But `upgradeConfig` builds one `engine.NewSchema` over every catalog section, which rejects duplicates. Reproduced: applets `a` and `b` each with `conf:"port"` — `./p3 a` exits 0, but `--upgrade-config` exits 2 with `duplicate argument --port` and `duplicate environment variable A__PORT`. Arg/env collisions are irrelevant to a file transform. No test covers cross-closure name reuse.

**3. MAJOR — main.go:380 — `--validate-config` runs user `Configured()` code, contradicting the spec's explicit promise.**
Spec §6: "runs the config layer only — never `Configured()`". But `execute` calls `pre := rt.prepareTranslator(p.res)` *before* branching on `p.validate`, and `prepareTranslator` runs Inject + `Configured()` over the Translator's whole dependency subtree. Reproduced: Translator whose Configured prints — `--validate-config` prints it, exit 0. The work is also useless in this mode. §7 grants the translator-first pass to `--help`/`--write-config` only. Untested (both `TestValidateConfig` worlds are translator-free).

**4. MINOR — introspect.go:170 — the `Arguments` best-effort fallback builds the schema under the caller's raw ref instead of the applet's primary alias, producing drifted/invalid env names.**
`plan()` and `helpSchema` use `primaryAlias(d)`; the fallback passes the untouched `appletID` parameter. Reproduced: `Arguments("example.com/probe", …)` returns `Env: "EXAMPLE.COM/PROBE__GREETING"` where execution truth is `PROBE__GREETING`. Also breaks the `svc.id == appletID` active-section positional binding. Spec: "introspection truth cannot drift" (§4). Untested.

**5. MINOR — main.go:379-391 — buffered startup logs are silently dropped on clean short-circuit runs; the engine's versionless-section warning never surfaces under `--write-config`.**
The buffer replays only in `lifecycle` or in `report` on violations. Reproduced: a normal run prints the versionless WARN; `--write-config` over the same file prints nothing — precisely the mode where the operator is normalizing that file. Same swallow for clean `--help` and `--validate-config`.

**6. MINOR — identity.go:79-86 (`validAlias`) — the separator-run ban is not enforced at the registration commit.**
`validAlias` accepts `a--b`. Reproduced: `.Alias("a--b")` commits and Builds cleanly, then *every invocation* dies with the engine's sep-run violation. For a config-less service the illegal alias is never caught at all. Same gap in `AppBuilder.Alias` validation (builder.go:205).

**7. MINOR — catalog.go:114-117 — the migration chain's shape is not validated at the `.Register()` commit; spec §6's "The commit validates the chain" sentence vs the IMPLEMENTED note placing validation in NewSchema — the spec is internally inconsistent; reconcile one or the other.**

**8. MINOR — main.go:271-281 (`sections`) — help sections render in dependency (resolution) order, not the Order-driven listing order the spec promises.** Usage listings and `Applets()` correctly follow composed order; help sections do not.

**9. NIT — internal/registry isValidID vs identity.go validServiceID — the two id grammars drift.**
`validServiceID` allows digit-first segments (`sxcli.dev/9lives` registers); `isValidID` rejects that id in an inject tag. Conversely `isValidID` accepts leading `_` which `validServiceID` refuses.

**10. NIT — introspect.go:80-85 — `SingleApplet` doc says it returns "its id"; the code returns the primary alias (correct — the comment is wrong). Introspector type comment: "reserved id `introspection`" — that's the alias; the id is `sxcli.dev/fw/introspection`.**

**11. NIT — main.go:109-122 + 671-683 — the alias/id pathological-tie failure double-reports** ("say which" + a misleading `unknown service` for the same ref).

**Test gaps:** Arguments+`--upgrade-config` inertness; validate-config's never-Configured promise; upgrade over cross-closure name reuse; fallback-Arguments env fidelity; fw-level assertions that cycle/unused-override warnings are actually emitted; the separator-run alias ban.

**Verified clean:** dispatch rules incl. System carve-out and `.exe` stripping; builder composition checks; graph tie-breaking incl. Order-sequence-among-ranked; Tarjan/SCC, Subtree, reverse-Stop, translator degradation, Tr/TrN; suppression mapping incl. from-version riding FeatureUpgradeConfig; Introspector squatting/reservation; ejection-unless-introspecting; SCM handler. README/CHANGELOG claims check out — except README's "never touches the other four applets' arguments" is falsified in `--upgrade-config` mode by finding 2.

## Overall assessment

Unusually spec-faithful: pipeline, composition model, identity split and visibility rules implement §4/§5 essentially verbatim; tests are dense; the error-message discipline is consistently executed. The defects cluster at the *newest* seams — the 2026-07-18/19 parity passes: the upgrade early-return leaves a nil schema the older Arguments path dereferences (the one crash), the whole-catalog upgrade schema reuses closure-scoped validation it shouldn't, and validate-config inherited the translator-first pass. All three are contained, mechanical fixes (guard the upgrade plan in Arguments; build the upgrade schema without arg/env registration or per-section; gate prepareTranslator on !p.validate), each deserving a pinning test.
